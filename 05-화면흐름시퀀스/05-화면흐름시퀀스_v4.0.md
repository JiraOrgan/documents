## 3. 시퀀스 다이어그램

### 3.1 회원가입 → 로그인 → 온보딩

```mermaid
sequenceDiagram
    actor U as 학습자
    participant W as Web/Flutter
    participant API as API Gateway
    participant Auth as 사용자 서비스
    participant DB as MySQL
    participant OB as 온보딩 서비스

    U->>W: 회원가입 폼 입력 (SCR-001)
    W->>API: POST /api/v1/auth/signup
    API->>Auth: 이메일 중복 확인
    Auth->>DB: SELECT users WHERE email=?
    DB-->>Auth: 결과
    alt 이메일 중복
        Auth-->>W: 409 Conflict
        W-->>U: "이미 사용 중인 이메일" 표시
    else 가입 성공
        Auth->>DB: INSERT users + user_profiles
        Auth-->>W: 201 Created
        W-->>U: 로그인 페이지로 이동 (SCR-002)
    end

    U->>W: 이메일/비밀번호 입력 (SCR-002)
    W->>API: POST /api/v1/auth/login
    API->>Auth: 인증 처리
    Auth->>DB: 비밀번호 검증
    DB-->>Auth: 사용자 정보 + 역할
    Auth-->>W: JWT Access Token + Refresh Token
    W-->>U: 역할 확인

    alt LEARNER & 온보딩 미완료
        W->>API: GET /api/v1/onboarding/status
        API->>OB: 온보딩 완료 여부 확인
        OB->>DB: SELECT diagnostic_results WHERE user_id=?
        DB-->>OB: null (미완료)
        OB-->>W: { completed: false }
        W-->>U: 온보딩 화면으로 이동 (SCR-003)
    else 온보딩 완료 or INSTRUCTOR/ADMIN
        W-->>U: 대시보드로 이동
    end
```

```mermaid
sequenceDiagram
```

### 3.3 AI 튜터 채팅 — SSE 스트리밍

```mermaid
sequenceDiagram
    actor U as 학습자
    participant W as Web/Flutter
    participant API as API Gateway
    participant AIGTW as AI Gateway
    participant PII as PII Masking Layer
    participant RAG as RAG Service
    participant LLM as LLM Service
    participant Cache as Semantic Cache

    U->>W: 질문 입력 → "전송" (SCR-104 사이드패널)
    W->>API: POST /api/v1/ai/tutor/chat (SSE 요청)
    API->>AIGTW: 메시지 + context (course_id, session_id)

    Note over AIGTW,PII: PII 입력 마스킹
    AIGTW->>PII: Input PII 스캔 (Regex + NER)
    PII-->>AIGTW: 마스킹된 메시지 (<NAME_1> 등)

    Note over AIGTW,Cache: Semantic Cache 확인
    AIGTW->>Cache: similarity > 0.95 검색
    alt 캐시 히트
        Cache-->>W: 캐시된 응답 반환 (SSE)
        W-->>U: 스트리밍 응답 표시
    else 캐시 미스
        AIGTW->>RAG: RAG 파이프라인 실행
        Note over RAG: Query Rewrite → Hybrid Search<br/>(pgvector + BM25) → Re-ranking<br/>→ Context Compression
        RAG-->>AIGTW: 압축된 컨텍스트 (Top 5 청크)

        AIGTW->>LLM: Prompt + 컨텍스트 + 메모리 + 사용자 질문
        Note over AIGTW,LLM: 레벨 분기 프롬프트<br/>Level1/2/3 (concept_mastery 기반)
        LLM-->>AIGTW: SSE 스트리밍 토큰

        loop SSE 청크 전송
            AIGTW->>PII: Output PII 스캔
            PII-->>AIGTW: 스캔된 청크
            AIGTW-->>W: data: {"token": "..."}
            W-->>U: 실시간 타이핑 효과 표시
        end

        AIGTW-->>W: data: [DONE]
        AIGTW->>Cache: 응답 Semantic Cache 저장
    end

    W-->>U: 추천 질문 3개 표시
    U->>W: 피드백 클릭 (👍 / 👎)
    W->>API: POST /api/v1/ai/feedback (message_id, feedback)
```

### 3.4 RAG 파이프라인 상세

```mermaid
sequenceDiagram
    participant EW as Embedding Worker
    participant SC as Semantic Chunking
    participant EMB as Embedding Service
    participant VDB as pgvector
    participant RAG as RAG Service
    participant ES as Elasticsearch
    participant RR as Re-ranker

    Note over EW,VDB: 인덱싱 흐름 (비동기 — ContentCreated 이벤트)
    EW->>SC: 강의 콘텐츠 청킹 요청
    SC->>SC: Recursive Splitter (200~400 토큰, 20% overlap)
    SC->>SC: Semantic Boundary Detection
    SC->>SC: chunk_hash(SHA-256) 계산
    SC-->>EW: 청크 목록

    loop 각 청크
        EW->>VDB: chunk_hash 중복 확인
        alt 동일 청크 (hash 일치)
            VDB-->>EW: SKIP (재임베딩 불필요)
        else 신규/변경 청크
            EW->>EMB: 임베딩 생성 (text-embedding-3-small)
            EMB-->>EW: vector(1536)
            EW->>VDB: INSERT content_embeddings (status=ACTIVE)
        end
    end

    Note over RAG,RR: 검색 흐름 (실시간)
    RAG->>RAG: Query Rewrite (LLM)
    par Hybrid Search
        RAG->>VDB: Vector Search Top-20 (course_id 격리)
    and
        RAG->>ES: BM25 Search Top-20
    end
    RAG->>RAG: RRF 융합 정렬 → Top-20
    RAG->>RR: CrossEncoder Re-ranking
    RR-->>RAG: Top-5 청크
    RAG->>RAG: Context Compression (토큰 40~60% 절감)
    RAG-->>RAG: 압축 컨텍스트 반환
```

### 3.5 퀴즈 제출 → AI 채점 → Confidence → HITL

```mermaid
sequenceDiagram
    actor U as 학습자
    participant W as Web
    participant API as API Gateway
    participant QS as 퀴즈 서비스
    participant OB as Outbox
    participant KF as Kafka
    participant AG as AI Grading Worker
    participant AI as AI Gateway
    participant MQ as Manual Review Queue
    actor I as 강사

    U->>W: 퀴즈 답변 제출 (SCR-106)
    W->>API: POST /api/v1/quiz/{quizId}/submit

    Note over QS,OB: Transactional Outbox
    QS->>QS: quiz_attempts INSERT
    QS->>OB: outbox_events INSERT (QuizSubmitted)
    QS-->>W: 202 Accepted

    OB->>KF: QuizSubmitted 이벤트 발행
    KF->>AG: 이벤트 수신 (dedup_key 체크)

    AG->>AI: AI 채점 요청 (rubric_coverage 포함)
    AI-->>AG: score + ai_feedback + rubric_coverage

    Note over AG: Confidence 계산<br/>= rubric_match*0.3 + answer_determinism*0.25<br/>+ score_consistency*0.25 + rubric_coverage*0.2

    alt Confidence ≥ 0.8
        AG->>QS: 채점 확정 (status=CONFIRMED)
        QS-->>W: 채점 결과 푸시
        W-->>U: 퀴즈 결과 표시 (SCR-107)
    else Confidence < 0.8
        AG->>MQ: Manual Review Queue 이관
        MQ-->>I: 검토 알림
        I->>W: Manual Review Queue 확인 (SCR-206)
        I->>API: PUT /api/v1/review/{submissionId}/grade
        API->>QS: 강사 최종 점수 저장
        QS-->>W: 채점 결과 푸시
        W-->>U: 최종 결과 표시 (SCR-107)
    end

    Note over AG: 채점 후 Analytics 업데이트
    AG->>KF: LessonCompleted 이벤트
    KF->>KF: Analytics Worker → concept_mastery 갱신
```

```mermaid
    participant NT as 알림 서비스
    actor I as 강사

    U->>W: 채점 결과 확인 (SCR-109)
    U->>W: "이의 제기" 클릭 (채점 후 7일 이내)
    W->>API: POST /api/v1/submissions/{id}/appeal\n{ appeal_reason: "..." }

    Note over AS,OB: Transactional Outbox
    AS->>AS: status = APPEALED, appeal_reason 저장
    AS->>OB: outbox_events INSERT (GradingAppeal)
    AS-->>W: 202 Accepted
    W-->>U: "이의 제기 접수 완료" 표시

    OB->>KF: GradingAppeal 이벤트
    KF->>NT: 알림 Consumer
    NT-->>I: "채점 이의 제기 접수" 알림

    I->>W: Manual Review Queue 확인 (SCR-206)
    I->>W: 이의 제기 항목 선택
    I->>W: 학습자 답변 + AI 채점 + 이의 사유 검토
    I->>API: PUT /api/v1/submissions/{id}/review\n{ final_score, feedback }
    API->>AS: 최종 점수 + 피드백 저장 (status=CONFIRMED)
    AS-->>W: 결과 업데이트
    NT-->>U: "채점 이의 제기 처리 완료" 알림
    W-->>U: 최종 결과 확인 가능
```

### 3.8 관리자 FinOps — Kill-switch 흐름

```mermaid
sequenceDiagram
    participant FO as FinOps Monitor
    participant DB as MySQL
    participant KF as Kafka
    participant NT as 알림 서비스
    participant KS as Kill-switch
    participant AIGTW as AI Gateway
    actor A as 관리자

    loop 5분마다 비용 집계
        FO->>DB: SELECT SUM(cost_usd) FROM ai_cost_logs WHERE DATE=TODAY
        DB-->>FO: current_usage_usd
    end

    alt current_usage > soft_limit ($80)
        FO->>KF: CostThresholdReached (SOFT)
        KF->>NT: 알림 Consumer
        NT-->>A: Slack/이메일 알림 "비용 Soft Limit 초과"
        AIGTW->>AIGTW: Opus 모델 비활성화
    end

    alt current_usage > hard_limit ($150)
        FO->>KF: CostThresholdReached (HARD)
        KF->>KS: Kill-switch 활성화
        KS->>DB: UPDATE cost_thresholds SET is_killed=true
        KS->>AIGTW: Haiku 전용 + 배치 중단
        NT-->>A: 긴급 알림 "Hard Limit — Kill-switch 활성화"
    end

    A->>W: Admin 대시보드 확인 (SCR-303 → SCR-305)
    A->>W: "Kill-switch 해제" 클릭
    W->>API: POST /api/v1/admin/finops/kill-switch/disable
    API->>KS: is_killed = false
    KS->>AIGTW: 정상 라우팅 복원
```
