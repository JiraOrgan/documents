## 1. 개요

### 1.1 목적

본 문서는 LearnFlow AI — AI 기반 적응형 학습 관리 시스템의 기능 요구사항(FR)과 비기능 요구사항(NFR)을 정의하고, 요구사항 추적 매트릭스(RTM)를 통해 화면 · API · 테스트와의 매핑을 제공한다.

### 1.2 용어 정의

| 용어 | 정의 |
|------|------|
| FR | Functional Requirement — 기능 요구사항 |
| NFR | Non-Functional Requirement — 비기능 요구사항 |
| RTM | Requirements Traceability Matrix — 요구사항 추적 매트릭스 |
| HITL | Human-in-the-Loop — 인간 개입 채점 프로세스 |
| RAG | Retrieval-Augmented Generation — 검색 증강 생성 |
| SSE | Server-Sent Events — 서버 단방향 스트리밍 |
| PII | Personally Identifiable Information — 개인 식별 정보 |
| OTel | OpenTelemetry — 분산 추적 표준 |
| RAGAS | Retrieval-Augmented Generation Assessment — RAG 자동 평가 |

### 1.3 우선순위 기준

| 우선순위 | 기준 |
|----------|------|
| P0 — 필수 | MVP에 포함, 미구현 시 서비스 불가 |
| P1 — 중요 | M2~M4 마일스톤 필수, 핵심 가치 제공 |
| P2 — 권장 | M5~M6 마일스톤, 품질/운영 향상 |
| P3 — 선택 | Phase 6 이후 또는 후속 스프린트 |

## 2. 기능 요구사항 (FR)

### 2.1 사용자 및 인증 (FR-AUTH)

| ID | 요구사항 | 우선순위 | 관련 API | Phase |
|----|----------|----------|----------|-------|
| FR-AUTH-01 | 시스템은 이메일/비밀번호 기반 회원가입을 제공해야 한다. 비밀번호는 bcrypt로 해싱 저장한다. | P0 | `POST /api/v1/auth/signup` | P1 |
| FR-AUTH-02 | 시스템은 로그인 성공 시 Access Token(15분) + Refresh Token(7일) JWT를 발급해야 한다. | P0 | `POST /api/v1/auth/login` | P1 |
| FR-AUTH-03 | 시스템은 Refresh Token을 통한 Access Token 갱신을 지원해야 한다. Refresh Token은 Redis에 저장하며 재사용 감지 시 즉시 무효화한다. | P0 | `POST /api/v1/auth/refresh` | P1 |
| FR-AUTH-04 | 시스템은 학습자(LEARNER) / 강사(INSTRUCTOR) / 관리자(ADMIN) 3개 역할(RBAC)을 지원해야 한다. | P0 | `GET /api/v1/users/me` | P1 |
| FR-AUTH-05 | 사용자는 닉네임, 아바타, 자기소개, 학습 선호도(속도, 일일 목표, 관심 분야, 난이도)를 수정할 수 있어야 한다. | P1 | `PUT /api/v1/users/me`, `PUT /api/v1/users/me/learning-prefs` | P1 |

| FR-COURSE-02 | 레슨은 텍스트(TipTap), 영상(URL), 파일 첨부 타입을 지원해야 한다. 파일은 MinIO/S3에 저장한다. | P0 | `POST /api/v1/sections/{id}/lessons` | P1 |

| FR-COURSE-04 | 학습자는 강의에 수강 신청할 수 있으며, 강의별 진도율(%)을 실시간으로 확인할 수 있어야 한다. | P0 | `POST /api/v1/courses/{id}/enroll`, `GET /api/v1/courses/{id}/progress` | P1 |
| FR-COURSE-05 | 학습자는 레슨 완료 처리를 할 수 있으며, `LessonCompleted` 이벤트가 발행되어 Analytics Worker가 진도/취약점을 갱신해야 한다. | P1 | `PUT /api/v1/lessons/{id}/complete` | P2 |
| FR-COURSE-06 | 비로그인 사용자도 강의 목록과 상세 정보(가격, 카테고리, 평점 포함)를 조회할 수 있어야 한다. | P0 | `GET /api/v1/courses`, `GET /api/v1/courses/{id}` | P1 |

### 2.3 AI 튜터 (FR-TUTOR)

| ID | 요구사항 | 우선순위 | 관련 API | Phase |
|----|----------|----------|----------|-------|
| FR-TUTOR-01 | 학습자는 강의별 AI 튜터 채팅 세션을 생성하고, 자연어 질문을 입력할 수 있어야 한다. | P0 | `POST /api/v1/ai/chat/sessions`, `POST /api/v1/ai/chat/sessions/{id}/messages` | P4 |
| FR-TUTOR-02 | AI 튜터 응답은 SSE(Server-Sent Events) 스트리밍으로 제공되어야 한다. 사용자는 응답이 완료되기 전에 텍스트를 점진적으로 확인할 수 있어야 한다. | P1 | `POST /api/v1/ai/chat/sessions/{id}/messages` (SSE) | P4 |
| FR-TUTOR-03 | AI 튜터는 학습자의 `concept_mastery` 평균 기반으로 3단계 레벨을 자동 분류하고 프롬프트를 분기해야 한다. Level 1(mastery < 0.4): 비유·그림 설명 / Level 2(0.4~0.7): 코드 예시·원리 / Level 3(≥ 0.7): 내부 구현·트레이드오프. | P1 | — (내부 로직) | P4 |
| FR-TUTOR-04 | AI 튜터는 세션 내 Short-term Memory(최근 10턴, Redis TTL 24h)와 학습자별 Long-term Memory(`concept_mastery`, `weakness_analyses`, MySQL)를 활용하여 문맥을 유지해야 한다. | P1 | — (내부 로직) | P4 |
| FR-TUTOR-05 | AI 튜터는 RAG 파이프라인을 통해 현재 수강 중인 강의 콘텐츠만을 검색 범위로 제한하여 답변을 생성해야 한다(`course_id` 기반 격리). | P1 | — (내부 로직) | P4 |
| FR-TUTOR-06 | 학습자는 AI 튜터 응답에 👍/👎 피드백을 남길 수 있어야 하며, 피드백은 `ai_feedback_logs`에 기록되어 3층 평가 Layer 1 신호로 활용되어야 한다. | P1 | `POST /api/v1/ai/chat/messages/{id}/feedback` | P4 |
| FR-TUTOR-07 | AI 튜터는 현재 레슨 주제 기반으로 추천 질문 3개를 생성하여 제공해야 한다. | P2 | `GET /api/v1/ai/chat/sessions/{id}/suggestions` | P4 |
| FR-TUTOR-08 | 학습자는 AI 튜터 대화 히스토리를 세션별로 조회할 수 있어야 한다. | P2 | `GET /api/v1/ai/chat/sessions`, `GET /api/v1/ai/chat/sessions/{id}/messages` | P4 |

### 2.4 RAG 파이프라인 (FR-RAG)

| ID | 요구사항 | 우선순위 | 관련 API | Phase |
|----|----------|----------|----------|-------|
| FR-RAG-01 | 콘텐츠 임베딩은 Semantic Chunking 하이브리드 방식으로 수행되어야 한다. Recursive Splitter(200~400 토큰, 20% overlap) + Semantic Boundary Detection(유사도 기반 병합/분리)을 조합한다. | P1 | — (Embedding Worker) | P4 |
| FR-RAG-02 | 각 청크는 SHA-256 `chunk_hash`를 계산하여 동일 내용의 재임베딩을 스킵해야 한다. 변경된 청크만 재임베딩하여 비용을 절감한다. | P1 | — (Embedding Worker) | P4 |
| FR-RAG-03 | 콘텐츠 삭제 시 해당 청크는 `status = INACTIVE`로 Soft Delete 처리되어야 한다. 90일 후 물리 삭제된다. | P1 | — (이벤트 `ContentDeleted`) | P4 |
| FR-RAG-04 | RAG 검색은 Vector Search(pgvector Top 20) + BM25(Elasticsearch Top 20)를 병렬 수행하고 RRF(Reciprocal Rank Fusion)로 융합하는 Hybrid Search를 사용해야 한다. | P1 | — (RAG Service) | P4 |
| FR-RAG-05 | RAG 검색 전 사용자 질문은 Query Rewrite를 통해 검색 최적화된 쿼리로 변환되어야 한다(예: "이거 왜 안됨?" → "JPA Lazy Loading 동작 원리"). | P1 | — (RAG Service) | P4 |
| FR-RAG-06 | Hybrid Search 결과는 CrossEncoder(ms-marco-MiniLM)를 통한 Re-ranking으로 Top 5를 선별해야 한다. | P1 | — (RAG Service) | P4 |
| FR-RAG-07 | Re-ranking 결과는 Context Compression을 통해 질문 관련 핵심 문장만 추출되어야 한다. 이를 통해 LLM 입력 토큰을 40~60% 절감한다. | P2 | — (RAG Service) | P4 |

### 2.5 퀴즈 및 과제 (FR-QUIZ)

| ID | 요구사항 | 우선순위 | 관련 API | Phase |
|----|----------|----------|----------|-------|
| FR-QUIZ-01 | 강사는 레슨을 지정하여 AI 자동 퀴즈 생성을 요청할 수 있어야 한다. 생성된 퀴즈는 `bloom_level`(기억/이해/적용/분석/평가)을 포함해야 한다. | P1 | `POST /api/v1/ai/quizzes/generate` | P4 |
| FR-QUIZ-02 | 학습자는 레슨별 퀴즈를 제출할 수 있어야 하며, `QuizSubmitted` 이벤트가 Outbox를 통해 AI Grading Worker로 전달되어야 한다. | P1 | `POST /api/v1/quizzes/{id}/submit` | P2 |
| FR-QUIZ-03 | AI 채점은 `rubric_coverage`(루브릭 항목 커버율, 가중치 0.2 포함) Confidence Score를 산출해야 한다. Confidence = weighted_avg(rubric_match_clarity×0.3 + answer_determinism×0.25 + score_consistency×0.25 + rubric_coverage×0.2). | P1 | — (AI Grading Worker) | P4 |
| FR-QUIZ-04 | Confidence ≥ 0.8인 채점은 자동 확정(CONFIRMED)되어야 하고, Confidence < 0.8인 채점은 Manual Review Queue에 자동 이관되어야 한다. | P1 | — (AI Grading Worker) | P4 |
| FR-QUIZ-05 | 학습자는 채점 완료 후 7일 이내 이의 제기(Appeal)를 할 수 있어야 하며, `GradingAppeal` 이벤트가 발행되어 강사 Manual Review Queue에 등록되어야 한다. | P1 | `POST /api/v1/quizzes/attempts/{id}/appeal`, `POST /api/v1/assignments/submissions/{id}/appeal` | P2 |
| FR-QUIZ-06 | 강사는 Manual Review Queue를 조회하고 채점을 최종 확정할 수 있어야 한다. | P1 | `GET /api/v1/instructor/review-queue`, `PUT /api/v1/instructor/review-queue/{id}/resolve` | P2 |
| FR-QUIZ-07 | 학습자는 퀴즈 결과와 AI 피드백을 조회할 수 있어야 한다. | P1 | `GET /api/v1/quizzes/{id}/results` | P2 |
| FR-QUIZ-08 | 과제는 자유 텍스트 제출을 지원하며, `AssignmentSubmitted` 이벤트를 통해 AI 채점이 수행되어야 한다. `ai_confidence` 컬럼에 신뢰도가 저장된다. | P1 | `POST /api/v1/assignments/{id}/submit` | P2 |

### 2.7 온보딩 (FR-ONBOARD)

| ID | 요구사항 | 우선순위 | 관련 API | Phase |
|----|----------|----------|----------|-------|
| FR-ONBOARD-01 | 시스템은 강의 최초 수강 시 Cold Start 해결을 위한 진단 테스트(3~5분, 5문항)를 제공해야 한다. 문항은 Bloom's Taxonomy 배분(기억/이해/적용/분석/평가)을 따른다. | P1 | `GET /api/v1/onboarding/diagnostic/{courseId}` | P2 |

| FR-ONBOARD-03 | 진단 테스트 대신 자가 진단(30초) 옵션을 제공해야 한다. 입문(0.1)/초급(0.3)/중급(0.5)/고급(0.7) 중 선택하며 `confidence_weight = 0.3`으로 설정된다. | P1 | `POST /api/v1/onboarding/self-assess` | P2 |
| FR-ONBOARD-04 | `confidence_weight`가 낮은 개념은 더 빠르게 퀴즈를 제안하여 mastery와 confidence를 동시에 갱신해야 한다. | P2 | — (내부 로직) | P5 |

### 2.9 AI 품질 관리 (FR-QUALITY)

| ID | 요구사항 | 우선순위 | 관련 API | Phase |
|----|----------|----------|----------|-------|
| FR-QUALITY-01 | 시스템은 3층 AI 품질 평가 체계를 구현해야 한다. Layer 1: 사용자 피드백(👍👎, 재설명 요청) / Layer 2: RAGAS(3회 중앙값) + DeepEval(G-Eval, Hallucination) + Importance Sampling / Layer 3: 인간 전문가 주간 리뷰. | P1 | `GET /api/v1/admin/ai/quality/dashboard` | P5 |
| FR-QUALITY-02 | RAGAS 자동 평가는 Faithfulness, Context Precision, Context Recall, Answer Relevancy를 측정하며, 동일 메시지에 대해 3회 평가 후 중앙값을 사용해야 한다. | P1 | `GET /api/v1/admin/ai/quality/ragas/summary` | P5 |
| FR-QUALITY-03 | DeepEval은 G-Eval과 Hallucination Score를 병행 측정하며, Faithfulness < 0.7인 응답은 자동으로 관리자 리뷰 리포트에 포함되어야 한다. | P1 | `GET /api/v1/admin/ai/quality/ragas/trends` | P5 |
