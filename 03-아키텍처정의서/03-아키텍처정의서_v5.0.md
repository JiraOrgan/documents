## 1. 아키텍처 개요

LearnFlow AI는 **Spring Boot 4 + React 18 + Flutter 3.x + LLM API** 기반의 AI 적응형 학습 관리 시스템이다.
주요 아키텍처 원칙:

- **이벤트 기반 비동기**: Transactional Outbox + Kafka (데이터 일관성 보장)
- **AI 서비스 분리**: AI Gateway Orchestrator + 4 전문 서브서비스
- **분산 추적**: OpenTelemetry Sampling (dev 100% / prod 10~30%, 에러 100%)
- **컴플라이언스**: PII 양방향 마스킹 (Input + Output), 감사 로그
- **FinOps**: Unit Economics 기반 동적 라우팅, Kill-switch

### 3.2 LLM 라우팅 — 4분류 + 예산 동적

| Tier | 모델 | 대상 요청 유형 | 예상 비용 |
|------|------|--------------|-----------|
| Tier 1 | Claude Haiku | 짧은 Q&A, OX, 용어 정의 | ~$0.001/요청 |
| Tier 2 | Claude Sonnet | 코드 설명, 디버깅, 리팩토링 | ~$0.005/요청 |
| Tier 3 | Claude Sonnet | 개념 설명, 비교 분석, 원리 | ~$0.010/요청 |
| Tier 4 | Claude Opus | 복잡한 설계, 아키텍처 질문 | ~$0.050/요청 |

**예산 기반 동적 라우팅 (v4.0)**:

```
잔여 > 70% : 정상 (4분류)
50~70%      : Tier 4 (Opus) 비활성
30~50%      : Haiku 우선
< 30%       : Haiku 전용 + 배치 중단
```

## 4. Transactional Outbox Pattern

### 4.1 Outbox 흐름

```mermaid
sequenceDiagram
    participant APP as 비즈니스 서비스
    participant DB as MySQL
    participant DEBEZIUM as Debezium CDC
    participant KAFKA as Kafka
    participant CONSUMER as Consumer

    Note over APP,DB: @Transactional — 하나의 DB 트랜잭션
    APP->>DB: ① 비즈니스 데이터 저장
    APP->>DB: ② outbox_events INSERT (dedup_key, destination_topic)
    Note over DB: 원자적 커밋 — 이벤트 유실 불가

    DB-->>DEBEZIUM: binlog 변경 감지 (CDC)
    Note over DEBEZIUM: Phase 1: Polling + ShedLock\nPhase 2: Debezium binlog (Scale 시)
    DEBEZIUM->>KAFKA: destination_topic별 라우팅 발행

    KAFKA->>CONSUMER: 이벤트 수신

    Note over CONSUMER: Holy Trinity 멱등성 검증
    CONSUMER->>CONSUMER: ① dedup_key 체크 (aggregate_id+event_type+version)
    CONSUMER->>CONSUMER: ② Version OCC (Read Model) / dedup_key (알림/AI)
    CONSUMER->>CONSUMER: ③ DB Upsert (멱등 Write)

    alt 성공
        CONSUMER->>DB: outbox_events.status = CONSUMED
    else 실패 3회
        CONSUMER->>KAFKA: DLQ 토픽 발행
    end
```

### 4.2 이벤트 목록 (8개)

| 이벤트 | Producer | destination_topic | Consumer | 처리 내용 |
|--------|----------|-------------------|----------|-----------|
| `ContentCreated` | 학습 서비스 (Outbox) | `content.created` | Embedding Worker | 청킹 + 임베딩 생성 |
| `ContentUpdated` | 학습 서비스 (Outbox) | `content.updated` | Embedding Worker | chunk_hash 비교 → 변경분만 재임베딩 |
| `ContentDeleted` | 학습 서비스 (Outbox) | `content.deleted` | Embedding Worker | 해당 청크 INACTIVE 처리 (Soft Delete) |
| `QuizSubmitted` | 퀴즈 서비스 (Outbox) | `quiz.submitted` | AI Grading Worker | 자동 채점 + Confidence Score |
| `AssignmentSubmitted` | 과제 서비스 (Outbox) | `assignment.submitted` | AI Grading Worker | 과제 AI 채점 + rubric_coverage |
| `GradingAppeal` | 퀴즈/과제 서비스 | `grading.appeal` | Notification Consumer | 강사 Manual Review Queue 알림 |
| `LessonCompleted` | 학습 서비스 (Outbox) | `lesson.completed` | Analytics Worker | 진도/취약점/mastery 갱신 |
| `CostThresholdReached` | FinOps Monitor | `finops.threshold` | Admin Notification | 비용 경고 + Kill-switch 트리거 |

### 4.3 DLQ 처리 흐름

```mermaid
flowchart LR
    OUTBOX["outbox_events\nstatus=PENDING"] --> RELAY["Outbox Relay\n(Debezium/Polling)"]
    RELAY -->|성공| KAFKA["Kafka\n(destination_topic)"]
    RELAY -->|실패 retry_count >= 5| DL["status=DEAD_LETTER\n관리자 알림"]

    KAFKA --> CONSUMER["Consumer"]
    CONSUMER -->|성공| DONE["처리 완료"]
    CONSUMER -->|실패 3회| DLQ["DLQ 토픽\n(*.dlq)"]
    DLQ --> ADMIN["Admin 수동 재처리"]
```

## 6. 보안 설계 (7 Layer)

```mermaid
flowchart TB
    REQ["외부 요청"] --> L1

    L1["Layer 1: 입력 필터링\n길이 제한 + 위험 패턴 감지\n(Prompt Injection 방어)"]
    L2["Layer 2: PII Masking\nInput + Output 양방향 스캔\n(Presidio + KoNLPy, 한국법 준수)"]
    L3["Layer 3: System Prompt 격리\n절대 노출 금지\n사용자 메시지와 물리적 분리"]
    L4["Layer 4: Output Validation\n점수 범위 / JSON 스키마 검증\n비정상 출력 차단"]
    L5["Layer 5: 데이터 격리\ncourse_id 기반 RAG 범위 제한\n타 수강 강의 접근 불가"]
    L6["Layer 6: Tool 사용 제한\nDB 직접 조회 차단\n외부 URL / 파일 시스템 접근 차단"]
    L7["Layer 7: FinOps Kill-switch\nSoft Limit $80/일 → 알림\nHard Limit $150/일 → 자동 다운그레이드"]

    L1 --> L2 --> L3 --> L4 --> L5 --> L6 --> L7 --> RESP["응답"]
```

### 6.1 PII Masking Pipeline 상세

```mermaid
flowchart LR
    INPUT["사용자 입력"] --> REGEX["Step1: Regex\n한국 특화\n이메일/전화/주민번호\n카드번호/계좌번호\n여권/운전면허/사업자등록"]
    REGEX --> NER["Step2: NER\nPresidio + KoNLPy\n인명/주소/기관명"]
    NER --> TOKEN["Step3: 토큰 치환\n'김철수' → NAME_1\n매핑: Redis session-scoped\nPseudonymization 옵션"]
    TOKEN --> LLM["LLM API 호출"]
    LLM --> DEMASK["Output Demasking\nNAME_1 → 원본 복원"]
    DEMASK --> SCAN["Output PII 스캔\nLLM 신규 생성 PII 탐지\n→ 마스킹 + audit_log\n'OUTPUT_PII_DETECTED'"]
    SCAN --> CLIENT["최종 응답"]
```
